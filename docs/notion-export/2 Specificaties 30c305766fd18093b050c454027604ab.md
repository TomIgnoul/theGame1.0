# 2. Specificaties

## 2.1 Functionele Vereisten (Baseline) [Structuur: User Flow]

### Flow 1 - Discover (kaart & filters)

- **FR-01** Het systeem moet POI’s tonen als pins op een interactieve kaart van Brussel.
- **FR-02** Het systeem moet filters aanbieden: War, Museum, Streetart, Food, Culture.
- **FR-03** Het systeem moet pins dynamisch filteren op basis van gekozen thema.
- **FR-04** Het systeem moet toelaten een POI te selecteren via pin/kaartlijst.

### Flow 2 - Decide (details + content)

- **FR-05** Het systeem moet een detailpagina tonen voor de geselecteerde POI.
- **FR-06** De detailpagina moet minimaal naam en locatie tonen (indien beschikbaar).
- **FR-07** Het systeem moet een AI-story kunnen tonen voor de geselecteerde POI.
- **FR-08** Het systeem moet praktische info tonen **alleen als die uit dataset komt**; anders “niet gevonden/onzeker”.

### Flow 3 - Do (route builder + route tonen)

- **FR-09** Het systeem moet user input vragen: thema + tijd en/of afstand (km).
- **FR-10** Het systeem moet op basis van input een wandelroute genereren met stops.
- **FR-11** Het systeem moet de route visualiseren als lijn op de kaart.
- **FR-12** Het systeem moet stops nummeren en in volgorde tonen.
- **FR-13** Het systeem moet een “Start wandeling” actie aanbieden.

### Data (seed / scope guard)

- **FR-14** Het systeem moet seed data laden uit openadata.brussels (of vooraf ingeladen export).
- **FR-15** Het systeem moet de MVP beperken tot 5 POI’s (hard cap).

### Route-stop Chat (AI in de wandeling)

- **FR-16** Het systeem moet bij **elke stop in de route** een chatbox tonen waarmee de gebruiker vragen kan stellen aan de AI over **die specifieke POI**.
- **FR-16a** De chatbox moet automatisch POI-context laden (naam, categorie, metadata) zodat antwoorden stop-specifiek zijn.

### Analytics & Admin Portal (post-MVP governed track)

- **FR-17** Het systeem moet analytics-events registreren voor: `route generated`, `route generation failed`, `route started`, `POI detail opened`, `filter applied`, `story generated`, `stop chat opened`, `stop chat sent`.
- **FR-18** Het systeem moet analytics-data opslaan in dedicated analytics-opslag, gescheiden van chatcontent.
- **FR-19** Het systeem moet analytics-data read-only ontsluiten via backend API voor het admin portal.
- **FR-20** Het admin portal moet KPI-cards tonen voor route generations, route starts, route failures, POI detail views en chat sends.
- **FR-21** Het admin portal moet tijdgebaseerde grafieken tonen voor route generations en route starts.
- **FR-22** Het admin portal moet breakdown-grafieken tonen per thema en per POI.
- **FR-23** Het admin portal moet filtering ondersteunen op datumrange en, waar van toepassing, op thema.
- **FR-24** Het admin portal moet loading-, empty- en error-states stabiel afhandelen.
- **FR-25** Het systeem moet het admin portal en analytics-endpoints beperken tot geautoriseerde admin/stakeholder toegang.
- **FR-26** Het analytics/admin portal moet read-only blijven.

---

## 2.2 Niet-functionele vereisten (Security/Privacy MVP)

- **NFR-S1 (Secrets)** Geen keys in frontend of Git; secrets via env/secret storage.
- **NFR-S2 (Data minimization)** Geen persistente chatlogs; logging anoniem en minimaal.
- **NFR-S4 (Abuse control)** Rate limiting op chat/route endpoint
- **NFR-S5 (Admin access control)** Analytics-endpoints en admin portal vereisen geautoriseerde admin/stakeholder toegang.
- **NFR-S6 (Analytics data minimization)** Analytics-opslag bevat geen chat bodies, geen AI-antwoorden, geen request bodies en geen directe PII.
- **NFR-P1 (Analytics read performance)** Decision Needed: er is nog geen goedgekeurde performance-threshold voor analytics reads/admin portal queries.
- **NFR-R1 (Analytics failure isolation)** Failures in analytics logging of analytics reads mogen kernflows niet blokkeren.
- **NFR-G1 (Governance/traceability)** Elke analytics/admin-portal PR moet requirements, Gherkin, DEC en traceability bijwerken.

# 2.3 Acceptatiecriteria (gherkin)

## Acceptatiecriteria uitgelegd

### Wat zijn acceptatiecriteria?

Acceptatiecriteria beschrijven **wanneer een feature “af” en “correct” is**. Ze vertalen requirements naar **testbare voorwaarden** zodat er geen discussie is over “werkt het?”.

### Waarom gebruiken we ze?

- maken requirements **ondubbelzinnig**
- vormen de basis voor **testen + demo-script**
- helpen scope bewaken: alleen wat aan criteria voldoet telt als “done”

### Format: Given / When / Then

- **Given** = startcontext (wat is al waar?)
- **When** = actie van gebruiker of systeemtrigger
- **Then** = verwacht resultaat (wat moet zichtbaar/meetbaar zijn)

### Belangrijkste regels

- **When is alleen actie** (geen verwachtingen).
- **Then is alleen output/resultaat** (zichtbaar, meetbaar).
- Voeg UI-indicatoren toe als bewijs (bv. “Stop 2 - POI naam” in chat header).
- Voor MVP: per kernflow minstens **1 succes-scenario + 1 fail-scenario**.

### Voorbeelden die we al hebben

- Chatbox per route-stop: chat opent bij stop 2 en toont duidelijk “Stop 2 - ”.
- Route genereren: route heeft min. 2 stops, wordt als lijn getekend, stops zijn genummerd, en “Start wandeling” verschijnt.
- Fail-scenario: als route niet mogelijk is → duidelijke melding, geen crash.

### Filters (FR-02/03)

**Scenario: Filter op één thema**

**Given** de kaart toont meerdere POI-pins met verschillende thema-categorieën.

**When** de gebruiker klikt op filter **“Museum”**.

**Then** toont de kaart enkel POI-pins met categorie **“Museum”**.

**And** alle niet-“Museum” pins verdwijnen van de kaart.

**And** de UI toont dat filter “Museum” actief is (visual indicator).

**And** de gebruiker kan het filter opnieuw uitschakelen zodat alle pins terug zichtbaar worden.

**And** boven de kaart verschijnt een actieve badge: **“Museum”**.

Fail/edge scenario (belangrijk)

**Scenario: Filter levert geen resultaten**

**Given** de kaart toont POI-pins.

**When** de gebruiker klikt op een filter waarvoor geen POI’s bestaan in de dataset.

**Then** toont het systeem een duidelijke boodschap (“Geen resultaten”) en blijft de app stabiel (geen crash).

### Story + praktische info (FR-05/07/08)

**Scenario: POI detailpagina openen en content tonen**

**Given** de kaart toont POI’s en de gebruiker ziet een POI-pin “X”.

**When** de gebruiker klikt op de POI-pin “X”.

**Then** opent het systeem de detailpagina van POI “X”.

**And** de detailpagina toont minimaal **naam** en **locatie/adres** (indien beschikbaar in dataset).

**And** de detailpagina toont een **AI-story** over POI “X”.

**And** praktische info wordt getoond **alleen als** die beschikbaar is in de dataset.

**And** onder de sectie **“Praktische info”** verschijnt het label **“Praktische info niet gevonden”** wanneer data ontbreekt.

**Fail/edge scenario (dataset ontbreekt praktische info)**

**Scenario: Praktische info ontbreekt in dataset**

**Given** de gebruiker opent de detailpagina van POI “X” waarvoor praktische info ontbreekt in de dataset.

**When** de detailpagina laadt.

**Then** toont het systeem geen verzonnen praktische info.

**And** het systeem toont een label/boodschap: **“Praktische info niet gevonden”** (of “Onzeker”).

**And** de app blijft volledig bruikbaar (story + route acties blijven werken).

## Route genereren + visualiseren (FR-09 t/m FR-13)

**Scenario: Route genereren en tonen op kaart**

**Given** de gebruiker is op het route builder scherm en er zijn POI’s beschikbaar in de gekozen thema-categorie.

**When** de gebruiker selecteert een thema, vult tijd en/of afstand (km) in, en klikt op **“Genereer route”**.

**Then** genereert het systeem een route met een lijst van stops (**minstens 2**) in een duidelijke volgorde.

**And** de route wordt getoond op de kaart als een lijn (polyline).

**And** elke stop wordt weergegeven als een genummerde marker (**Stop 1, Stop 2, …**).

**And** de route-lijst en kaart zijn consistent (stopnummering en volgorde matchen).

**And** de gebruiker ziet een **“Start wandeling”** actie na succesvolle routegeneratie.

**Scenario: Geen route mogelijk binnen criteria**

**Given** de gebruiker is op het route builder scherm.

**When** de gebruiker klikt op **“Genereer route”** met criteria waarvoor geen route kan worden samengesteld.

**Then** toont het systeem een duidelijke boodschap (bv. “Geen route mogelijk binnen deze criteria”).

**And** het systeem geeft een suggestie (bv. “verhoog tijd/afstand” of “kies ander thema”).

**And** de app blijft stabiel (geen crash, gebruiker kan criteria aanpassen en opnieuw proberen).

### Chat per route-stop (FR-16/16a)

**Scenario: Chatbox openen voor een specifieke route-stop**

**Given** de gebruiker heeft een route gegenereerd en ziet **stop 2** in de route-lijst.

**When** de gebruiker klikt op **“Chat over deze stop”** bij stop 2.

**Then** opent het systeem een chatbox gekoppeld aan **stop 2**.

**And** bovenaan de chatbox staat een duidelijke header: **“Stop 2 — <POI naam>”**.

**And** de AI heeft automatisch POI-context van stop 2 geladen (naam + categorie/metadata) zodat antwoorden stop-specifiek zijn.

**Scenario: Chat blijft bruikbaar bij ontbrekende POI-context**

**Given** de gebruiker opent de chatbox bij een stop waarvan sommige metadata ontbreekt.

**When** de gebruiker stelt een vraag in de chat.

**Then** antwoordt de AI met de beschikbare context zonder info te verzinnen.

**And** ontbrekende info wordt expliciet aangeduid (bv. “niet gevonden/onzeker”).

**And** de app blijft stabiel en de gebruiker kan verder wandelen/navigeren.

### Analytics & admin portal (GH-AN-01 t/m GH-AN-07)

**GH-AN-01 Scenario: Geautoriseerde admin ziet KPI-cards en grafieken**

**Given** een geautoriseerde admin/stakeholder heeft een geldige admin-sessie.

**And** er zijn analytics-records beschikbaar voor de gekozen periode.

**When** de admin het analytics portal opent.

**Then** toont het systeem KPI-cards voor route generations, route starts, route failures, POI detail views en chat sends.

**And** toont het systeem tijdgebaseerde grafieken voor route generations en route starts.

**And** toont het systeem breakdown-grafieken per thema en per POI.

**And** alle getoonde analytics-data is read-only.

**GH-AN-02 Scenario: Ongeautoriseerde toegang wordt geblokkeerd**

**Given** een gebruiker heeft geen geldige admin-sessie.

**When** de gebruiker het analytics portal opent of een analytics-endpoint oproept.

**Then** weigert het systeem de toegang.

**And** toont het systeem geen analytics-data.

**And** blijft het systeem stabiel zonder gevoelige details te lekken.

**GH-AN-03 Scenario: Datumrange-filter ververst metrics consistent**

**Given** een geautoriseerde admin/stakeholder bekijkt het analytics portal.

**And** er zijn analytics-records beschikbaar over meerdere datums en thema’s.

**When** de admin een datumrange kiest en, waar van toepassing, een themafilter activeert.

**Then** verversen KPI-cards, tijdgebaseerde grafieken en breakdown-grafieken op basis van dezelfde filterset.

**And** tonen alle widgets consistente aantallen voor dezelfde datasetselectie.

**GH-AN-04 Scenario: Lege analytics-dataset toont stabiele empty state**

**Given** een geautoriseerde admin/stakeholder opent het analytics portal voor een periode zonder analytics-records.

**When** het portal laadt.

**Then** toont het systeem een duidelijke empty state voor KPI-cards en grafieken.

**And** blijft de UI stabiel en read-only.

**And** wordt een lege dataset niet als backend error gepresenteerd.

**GH-AN-05 Scenario: Route success/fail/start maken analytics-records**

**Given** analytics-logging is ingeschakeld en analytics-opslag is beschikbaar.

**When** een gebruiker een route succesvol genereert, een routegeneratie faalt, of op **“Start wandeling”** klikt.

**Then** schrijft het systeem respectievelijk `route generated`, `route generation failed` en `route started` events weg in analytics-opslag.

**And** bevatten die analytics-records alleen de velden die nodig zijn voor tellingen en aggregaties.

**GH-AN-06 Scenario: Story/chat interacties worden geteld zonder chatcontent op te slaan**

**Given** analytics-logging is ingeschakeld.

**When** een gebruiker een POI-detail opent, een filter toepast, een story genereert, een stop-chat opent, of een stop-chatbericht verstuurt.

**Then** registreert het systeem respectievelijk `POI detail opened`, `filter applied`, `story generated`, `stop chat opened` en `stop chat sent` events in analytics-opslag.

**And** bevat analytics-opslag geen chat bodies, geen AI-antwoorden, geen request bodies en geen directe PII.

**GH-AN-07 Scenario: Analytics write failure breekt de primaire user flow niet**

**Given** een primaire user flow is geldig en de analytics-write mislukt.

**When** de gebruiker een actie uitvoert die normaal een analytics-event zou schrijven.

**Then** blijft de primaire flow werken volgens het kernresultaat van die actie.

**And** blokkeert de analytics-fout de gebruiker niet.

**And** blijft het analytics/admin portal een gescheiden, post-MVP governed track.
