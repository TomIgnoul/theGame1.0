## Phase 3 — Design (Overview)

This project follows a structured lifecycle. Phase 3 (“Design”) converts requirements into buildable architecture, platform, security, and UI specifications.

### 3.1 Architecture (Modules 1–5)

- Selected a **Frontend + Backend API (BFF-style)** architecture.
- Backend is the single gateway to external services (Maps/LLM) to keep secrets server-side and centralize controls.
- Defined core components: Frontend (UI), Backend API (FastAPI), Database (PostgreSQL), External APIs (Maps/LLM).

### 3.2 Database Design (PostgreSQL)

- Chosen database: **PostgreSQL**.
- Data quality strategy: **one DB with two schemas**:
    - `raw` = ingested “dirty” open data
    - `app` = sanitized canonical data used by the backend
- ETL strategy: **Python ETL** (scripts) to sanitize `raw → app`.
- Theme storage: **TEXT + CHECK constraint** (avoids ENUM migration friction).
- Routes storage: **hybrid**:
    - `app.route_stops` normalized (queryable for analytics)
    - `app.routes.coordinates` stored as JSONB (lighter than full point normalization)
- Minimal indexing for MVP (theme filter + basic geo index on `lat,lng`).

### 3.3 Platform Design (Local-first, Agentic-ready)

- **Local-first** (no cloud) to avoid costs and reduce friction.
- Docker Compose runtime includes:
    - `postgres` (DB)
    - `backend` (FastAPI)
    - `n8n` (automation/orchestration)
- Frontend runs locally (hot reload) and calls the backend at `http://localhost:8000`.
- **Scripts are canonical**, n8n triggers scripts:
    - `scripts/migrate.sh` (DB schema migrations)
    - `scripts/seed.sh` (ETL/seed raw → app)
- n8n workflows are **versioned in the repo**: `n8n/workflows/*.json`.
- MCP is included as the **agent tooling interface layer**:
    - config in `mcp/`
    - documentation in `docs/mcp/`

### 3.4 Security Design (MVP, No-login)

- MVP runs **without login**; security relies on API hardening:
    - **Secrets server-side** only (no keys in frontend or Git), **fail-fast** backend startup if keys missing.
    - **Rate limiting** in FastAPI (in-memory counters for MVP) on cost-sensitive endpoints:
        - `POST /routes/generate`
        - `POST /chat/stop`
    - **Data minimization**: no persistent chat logs; metrics-only logging (status, latency, error type).
    - Input validation including **max chat message length (500 chars)**.

### 3.5 UI Design (Mobile-first)

- UI pattern: **single-page map** with overlays:
    - POI Detail: full-screen
    - Route Builder: bottom sheet/panel
    - Chat: bubble + drawer overlay
- Defined per-screen state machines (loading/empty/error) and component checklists.
- Key UI proof points for acceptance:
    - filter badge visible
    - route renders as polyline + numbered stops
    - chat header shows `Stop {n} — {POI name}`
    - practical info never hallucinated; fallback label shown when missing

### 3.6 UI ↔ API Mapping

- Mapped UI actions to backend endpoints and error handling rules (400/404/429/5xx).
- Storytelling is handled via `POST /chat/stop` in “story mode” and is **button-triggered** (no auto-load).

### 3.7 API Contracts (Design-level)

- Defined endpoints and minimal request/response contracts:
    - `GET /health`
    - `GET /pois`, `GET /pois/{id}`
    - `POST /routes/generate`, `POST /routes/{route_id}/start`, (optional) `GET /routes/{route_id}`
    - `POST /chat/stop`
- Standardized error shape and validation rules.

### 3.8 Decision Log (DEC)

- Documented key design decisions (why + impact), including:
    - Local-first platform, BFF architecture, Postgres + raw/app schemas, Python ETL,
    - TEXT+CHECK for themes, hybrid routes storage,
    - No-login MVP, FastAPI in-memory rate limiting,
    - Story via chat endpoint, mobile overlay UI,
    - scripts canonical + n8n orchestration, workflows versioned in repo, MCP layering.
