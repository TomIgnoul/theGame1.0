# 3.8 Decicion log

## Doel

Dit log documenteert de belangrijkste architectuur- en ontwerpkeuzes, met motivatie en impact. Dit is belangrijk voor evaluatie (“waarom”) en om scope-creep te vermijden na sign-off.

---

## DEC-2026-001 - Local-first platform (geen cloud)

**Decision**: Local-first development, cloud out-of-scope.

**Waarom**: cloud kost geld en verhoogt setup-complexiteit. Docker Compose maakt lokale setup reproduceerbaar.

**Impact**: team draait identieke stack op eigen laptop; deployment later als post-MVP.

---

## DEC-2026-002 - Architectuur: Frontend + Backend API (BFF-stijl)

**Decision**: Frontend praat enkel met backend; backend praat met Maps/LLM.

**Waarom**: secrets server-side houden, rate limiting centraal, consistente business logic.

**Impact**: extra backend werk, maar lagere security- en abuse-risico’s.

---

## DEC-2026-003 - Database keuze: PostgreSQL

**Decision**: PostgreSQL als centrale datastore.

**Waarom**: stabiel, relational queries, goed voor analytics (routes/stops).

**Impact**: schema/migrations nodig, maar hoge betrouwbaarheid.

---

## DEC-2026-004 - Data quality aanpak: 1 DB, 2 schema’s (`raw` + `app`)

**Decision**: “dirty vs clean” scheiden via schema’s i.p.v. 2 aparte databases.

**Waarom**: zelfde conceptueel voordeel, minder infra en minder sync-failure modes.

**Impact**: ETL flow vereist (raw → app), maar eenvoudiger beheer.

---

## DEC-2026-005 - ETL/sanitisatie in Python

**Decision**: Sanitisatie gebeurt via Python ETL scripts.

**Waarom**: mapping/cleaning is leesbaar, logbaar, makkelijk uitbreidbaar.

**Impact**: extra script onderhouden; scripts zijn canoniek.

---

## DEC-2026-006 - Theme opslag als TEXT + CHECK (geen ENUM)

**Decision**: `theme` is TEXT met CHECK constraint.

**Waarom**: strict validation zonder ENUM-migratie-frictie bij wijziging.

**Impact**: schema blijft flexibel, maar whitelist moet in DB + backend synchroon.

---

## DEC-2026-007 - Routes opslag: hybride (stops genormaliseerd, coordinates JSONB)

**Decision**: `route_stops` tabel + `routes.coordinates` als JSONB.

**Waarom**: stops zijn waardevol voor stats/joins; points normaliseren is te zwaar voor MVP.

**Impact**: analytics op stops makkelijk; route render blijft simpel.

---

## DEC-2026-008 - No-login MVP + API hardening

**Decision**: geen login voor MVP; security via secrets + rate limiting + validation.

**Waarom**: scope beperken; toch kosten/misbruik beheersen.

**Impact**: IP-based controls zijn key; later kan auth toegevoegd worden.

---

## DEC-2026-009 - Rate limiting in FastAPI (in-memory)

**Decision**: rate limiting in FastAPI, counters in memory.

**Waarom**: MVP eenvoudig, geen extra infra (Redis).

**Impact**: reset bij restart; niet schaalbaar multi-instance (post-MVP).

---

## DEC-2026-010 - Storytelling via `/chat/stop`, button-load

**Decision**: story generation via hetzelfde chat endpoint; story laadt pas na button click.

**Waarom**: minder endpoints, 1 AI gateway, minder calls.

**Impact**: consistente AI control; UX blijft voorspelbaar.

---

## DEC-2026-011 - UI pattern: single-page map + overlays

**Decision**: map als base screen met overlays (detail full-screen, builder panel, chat drawer).

**Waarom**: mobile-first, context blijft zichtbaar, snelle navigation.

**Impact**: state management belangrijk, maar UX robuust.

---

## DEC-2026-012 - Scripts canoniek, n8n triggert scripts

**Decision**: migrations/seed via scripts; n8n orkestreert.

**Waarom**: scripts zijn reviewbaar en reproduceerbaar; n8n geeft agentic automation zonder lock-in.

**Impact**: beste van beide: reliability + “mindblowing” automation.

---

## DEC-2026-013 - n8n workflows versioned in repo

**Decision**: workflows exporteren naar `n8n/workflows/*.json`.

**Waarom**: versiebeheer, team reproduceerbaarheid, minder wrijving.

**Impact**: discipline: export + commit bij wijzigingen.

---

## DEC-2026-014 - MCP als tooling interface layer

**Decision**: MCP config in `mcp/`, docs in `docs/mcp/`.

**Waarom**: agents krijgen gecontroleerde tool access; least privilege; consistent gedrag.

**Impact**: agentic platform is structureel en auditeerbaar.

---

## DEC-2026-015 - Post-MVP analytics/admin portal track + backend-issued admin session

**Decision**: analytics/admin portal wordt een post-MVP governed delivery track. v1 admin access model = backend-issued admin-sessie via gedeelde env-passphrase + httpOnly cookie.

**Waarom**: het huidige systeem heeft geen login/accounts en secrets moeten server-side blijven. Dit laat admin/stakeholder toegang toe zonder frontend secrets of volwaardige account-flow in MVP-scope te introduceren.

**Impact**: analytics-endpoints en admin portal blijven read-only, vereisen server-side toegangscontrole, en horen bij een aparte post-MVP delivery track met verplichte DEC- en traceability-updates per analytics PR.
