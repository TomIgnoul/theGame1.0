# Theoretische onderbouwing (waarom deze architectuur logisch is)

## 1) Waarom een Backend-for-Frontend (BFF) laag?

Een BFF zit tussen frontend en externe/back-end services om client-specifieke noden af te handelen (zoals aggregatie, security, shaping). Dit past bij jullie case omdat de frontend **niet** rechtstreeks met Maps/LLM moet praten als je secrets en misbruik wil controleren.

## 2) Waarom secrets nooit in de frontend?

API keys in client code kunnen uitlekken via source control, bundling of network inspection. Best practice is keys **niet embedden in code** en ze server-side beheren (env vars/secret storage), plus key restrictions waar mogelijk.

## 3) Waarom rate limiting een must is

Rate limiting is een standaardmaatregel om API misbruik en kostenexplosie te voorkomen (zeker bij betaalde externe APIs zoals Maps/LLM). Dit is ook een veelgenoemde controle in API security guidance.

## 4) Waarom Node.js + Express voor deze backend

Node.js + Express sluit aan bij de daadwerkelijke implementatie en bij de tooling in `docs/mcd.md` (bijvoorbeeld gebruik van Prisma en JavaScript/TypeScript-ecosysteem). Dit maakt het ontwerp realistischer en verkleint het risico op ‚Äútelephone game‚Äù tussen ontwerp en code.
