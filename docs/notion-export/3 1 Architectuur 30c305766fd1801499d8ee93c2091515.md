# 3.1 Architectuur

## Doel en scope

We ontwerpen de MVP “Hidden Gems Brussels” als een **webapp** die:

- POI’s toont op een kaart + filters
- routes genereert op thema + tijd/km
- per route-stop chat met AI aanbiedt
- voldoet aan NFR’s: **secrets server-side, geen chatlogs, rate limiting**

## Module 1 - Architectuurkeuze

**Keuze:** **Frontend (React) + Backend API (Node.js + Express, BFF-stijl)**

**Waarom:** NFR’s vereisen centrale controle over secrets en abuse control; daarnaast maakt dit integratie met externe services (Maps/LLM) beheersbaar.

## Module 2 - Componenten (high-level)

- **Frontend (Web app)**
    - UI: kaart, filters, detail, route, chat
    - Geen API keys in client
- **Backend API (Node.js + Express)**
    - Gem endpoints
    - Route orchestration (Google Routes API server-side)
    - Story/chat orchestration (LLM calls server-side)
    - Rate limiting + minimale logging (geen chat opslag)
    - Admin auth via gedeelde env-passphrase + backend-issued httpOnly cookie
    - Read-only analytics endpoints voor overview, timeseries en breakdowns
- **Datastore (PostgreSQL)**
    - `datasets` (dataset provenance)
    - `gems` (genormaliseerde POI records)
    - `gem_stories` (AI story cache)
    - `route_logs` (optionele debug/analytics voor route-aanvragen)
    - `analytics_events` (dedicated analytics storage voor admin portal aggregaties)
    - **Geen chat messages** in storage
- **Externe services**
    - Open Data Brussels
    - Google Maps JavaScript API (frontend map rendering)
    - Google Routes API (backend routing)
    - LLM (GPT)

## Module 3 - API contracts (MVP, high-level overzicht)

De gedetailleerde contracten staan in **3.7 API Contracts**. Hier enkel de hoofdgroepen, afgestemd op de feitelijke app-implementatie en `docs/mcd.md`:

- **Gems**
    - `GET /api/gems` (optioneel filter: `theme`)
    - `GET /api/gems/:id`
- **Routes**
    - `POST /api/routes`
- **Stories**
    - `POST /api/gems/:id/story`
- **Analytics ingest**
    - `POST /api/analytics/events`
- **Admin auth + analytics (read-only)**
    - `POST /api/admin/auth/login`
    - `POST /api/admin/auth/logout`
    - `GET /api/admin/analytics/overview`
    - `GET /api/admin/analytics/timeseries`
    - `GET /api/admin/analytics/breakdowns`
- **Admin / data sync**
    - `POST /api/admin/datasets/sync`
    

## Module 4 - Data model (MVP)

**POI minimum schema**

- must: `id, name, lat, lng, theme, short_description`
- nice-to-have: `practical_info` (mag ontbreken → UI label “niet gevonden/onzeker”)

**Belangrijk:** routes zijn **stateless** in de MVP; alleen optionele `route_logs` worden bewaard voor debugging/analytics. Chat messages worden **niet** opgeslagen (NFR-S2).

Voor de post-MVP analytics/admin portal track worden alleen dedicated `analytics_events` opgeslagen. Read-side analytics blijft read-only en vereist een backend-issued admin-sessie via httpOnly cookie.

## Module 5 - Routing & chat flow decisions

**Routing via backend (A)**

- backend houdt Maps key secret
- backend kan rate limiting toepassen
- backend bewaakt consistentie van stops/constraints

**Polyline format**

- gekozen: **lijst van coördinaten** (makkelijk renderen/debuggen voor MVP)

**Chat context**

- gekozen: **frontend stuurt poi_id**, backend haalt POI-context uit datastore en bouwt prompt
    
    (single source of truth + minder tamper risk)

# Theoretische onderbouwing (waarom deze architectuur logisch is)

## 1) Waarom een Backend-for-Frontend (BFF) laag?

Een BFF zit tussen frontend en externe/back-end services om client-specifieke noden af te handelen (zoals aggregatie, security, shaping). Dit past bij jullie case omdat de frontend **niet** rechtstreeks met Maps/LLM moet praten als je secrets en misbruik wil controleren.

## 2) Waarom secrets nooit in de frontend?

API keys in client code kunnen uitlekken via source control, bundling of network inspection. Best practice is keys **niet embedden in code** en ze server-side beheren (env vars/secret storage), plus key restrictions waar mogelijk.

## 3) Waarom rate limiting een must is

Rate limiting is een standaardmaatregel om API misbruik en kostenexplosie te voorkomen (zeker bij betaalde externe APIs zoals Maps/LLM). Dit is ook een veelgenoemde controle in API security guidance.

## 4) Waarom Node.js + Express voor deze backend

Node.js + Express sluit aan bij de daadwerkelijke implementatie en bij de tooling in `docs/mcd.md` (bijvoorbeeld gebruik van Prisma en JavaScript/TypeScript-ecosysteem). Dit maakt het ontwerp realistischer en verkleint het risico op “telephone game” tussen ontwerp en code.

# “Waarom” document (Decision Log mini)

- **We kiezen BFF (frontend + backend)** omdat secrets + rate limiting centraal moeten → **dit betekent**: externe calls lopen via backend.
- **We doen routing via backend** omdat Maps keys server-side moeten blijven en we kosten/misbruik willen beheersen → **dit betekent**: frontend krijgt enkel route data terug.
- **We bewaren geen chatlogs** omdat data-minimization en privacy-by-design → **dit betekent**: chat is stateless request/response.
- **We slaan routes wél op** om heropening + simpele analytics mogelijk te maken → **dit betekent**: `routes` tabel/collectie + `started_count`.
- **We gebruiken polyline als coordinates[]** omdat het MVP-debuggen en UI implementatie versnelt → **dit betekent**: iets grotere payload, minder complexiteit.
